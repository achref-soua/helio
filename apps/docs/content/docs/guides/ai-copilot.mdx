Helio is AI-native, not AI-bolted-on. The copilot turns intent into working
drafts, and a set of predictive models quietly improve every send — all grounded
in **your own organization's data**.

## What the copilot does

| Capability                 | Prompt example                                                    |
| -------------------------- | ----------------------------------------------------------------- |
| Natural language → segment | "Trial users who opened an email last week but didn't convert."   |
| Natural language → journey | "A 5-step trial-conversion sequence with a wait and an A/B test." |
| Natural language → email   | "A friendly win-back email offering 20% off, on brand."           |

Every result is a **validated draft** you review and edit before anything runs.
The copilot proposes; you decide. It never sends, launches, or mutates on its
own.

## Predictive intelligence

| Model                   | What it produces                                         |
| ----------------------- | -------------------------------------------------------- |
| Predictive lead scoring | `conversionProbability` per contact, usable in segments. |
| Churn prediction        | `churnRisk` per contact.                                 |
| Send-time optimization  | The best hour to reach each contact.                     |
| Autonomous A/B winner   | Picks and promotes the winning variant via a z-test.     |

Scoring blends a gradient-boosted model over ClickHouse behavioral features
with a heuristic fallback, so it produces sensible results even with little
history. Predictions surface as contact fields and as
[segment](/docs/guides/segments) conditions.

## Grounding & security

The copilot's retrieval is **strictly scoped to your organization** over the
RLS-bound application role — the model chooses which tools to call, but it can
never widen its own scope to another tenant. Provider keys are read as secrets
and never logged.

## Bring your own AI — per organization

Each organization can connect its **own** AI provider under
**Settings → Provider credentials → AI copilot**: pick `openai`,
`anthropic`, `groq`, `ollama`, or any OpenAI-compatible `local` server
(vLLM, LM Studio, llama.cpp — enter its base URL), set the model, and paste
the key (local servers need none). The key is sealed with AES-256-GCM in
the [credential vault](/docs/security/credential-vault), shown only as a
mask, decrypted at call time, and never logged. **Verify** runs a read-only
models-list probe so you know the key works before the copilot uses it.
The copilot page shows which provider is serving your organization — and
when no provider is configured anywhere (no org credential, no deployment
key), it says so plainly and links to Settings instead of advertising a
model it cannot reach.

For full data sovereignty, point at a local model — nothing leaves your
infrastructure; prompts are refused over plaintext HTTP to remote hosts
either way.

Deployments can still set `INTEL_LLM_*` environment variables as the
fallback for organizations that haven't connected anything — see the
[configuration reference](/docs/configuration#intelligence--ai).

## Conversion events

Predictive scoring's positive label — which event names count as a
conversion — is configurable **per workspace** under
**Settings → Workspace analytics** (for example `Order Completed,
Signed Up`). Workspaces without their own list use the deployment default.

## For external agents

Everything above is also exposed to external AI agents through the
[MCP server](/docs/api/mcp), so an agent in another tool can search contacts,
draft a segment or journey, and read a workspace summary — within the same
tenant scope.
