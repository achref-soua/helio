Helio's analytics read from **ClickHouse**, the event store, so aggregations
over millions of events stay fast.

## The overview dashboard

The dashboard surfaces the health of your messaging at a glance:

- An **engagement timeline** of opens and clicks over time.
- **Per-campaign** breakdowns of sends, opens, and clicks.
- Headline contact and list counts.

<Callout type="info">
  Analytics **degrade gracefully**. If the full profile (and therefore ClickHouse) isn't running,
  the dashboard renders what it can from PostgreSQL instead of erroring — handy when you're working
  in the lightweight `core` profile.
</Callout>

## What's measured

| Metric                  | Source                                                        |
| ----------------------- | ------------------------------------------------------------- |
| Opens / clicks          | The [tracking service](/docs/guides/email) pixel & redirector |
| Campaign engagement     | Per-send rows joined with engagement events                   |
| A/B variant performance | Variant stamped on each send and event                        |
| Conversions             | Events you designate as conversions                           |

## Insights: funnels & retention

The **Insights** page runs two behavioral reports over the event stream. Both
read from ClickHouse, so they need the `full` Compose profile; without it they
show a "connect the analytics store" note rather than erroring.

### Conversion funnels

Enter an ordered list of event names — for example `Viewed Pricing, Signed Up,
Activated` — and a conversion window. Helio computes, per person, how far they
got through the sequence (using ClickHouse `windowFunnel`) and shows how many
reached each step, the conversion rate from the first step, and the drop-off
between steps. A person is identified by `user_id`, falling back to their
anonymous id, so the funnel follows them across an identify.

### Cohort retention

Helio groups people by the week they were first seen and shows the share still
active in each later week. Pick how many weeks to analyze (4–26). Period 0 is the
cohort size; each later cell is that cohort's retention, shaded by strength.

### Multi-touch attribution

Name the event that counts as a conversion (for example `Order Completed`).
Helio finds everyone who fired it, assembles the campaign touches that preceded
each person's first conversion, and distributes credit by the model you pick:

- **First touch** — all credit to the earliest campaign that touched them.
- **Last touch** — all credit to the most recent campaign before converting.
- **Linear** — credit split equally across the distinct campaigns involved.

Credit is summed across all converters and ranked by campaign, so you can see
which campaigns actually drive conversions under each model.

## SQL explorer

For anything the canned reports don't cover, the **SQL explorer** runs ad-hoc
read-only SQL over your events. It's deliberately fenced in, with defense in
depth:

- **Read-only** — the query must be a single `SELECT` (or `WITH … SELECT`); no
  comments, no quoted identifiers, and ClickHouse runs it with `readonly=1`.
- **Events only** — every `FROM`/`JOIN` target must be the `events` table or a
  CTE you define, which blocks `system.*`, other databases, and table functions
  (`remote`, `url`, `file`, `s3`, …). IO functions are blocked in any position.
- **Auto-scoped** — `events` is rewritten to a CTE pre-filtered to your
  workspace, so you only ever see your own rows. The workspace is also confirmed
  to belong to your organization before the query runs.
- **Capped** — results are limited to 1,000 rows with a 10-second execution
  budget.

So `SELECT event, count() FROM events GROUP BY event ORDER BY 2 DESC` just works,
while `SELECT * FROM system.tables` or `DROP TABLE events` are refused. Editor
role or above is required.
