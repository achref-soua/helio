Helio is **polyglot by design**. TypeScript owns the product surface — the
dashboard, the public API, and the journey workers that run on Temporal for
durable execution. Python owns the intelligence plane — scoring, content
generation, segment compute, and the MCP server. PostgreSQL holds transactional
state with row-level security per tenant; ClickHouse holds the event firehose
for analytics; Redpanda is the backbone between them.

| ----------------------- | --------------------------------------------------------------------------- |
| Marketing operator      | Builds segments, journeys, and campaigns via the dashboard and API.         |
| Tracked end-user        | Receives messages; generates events via the tracking SDK; opens and clicks. |

## Trust boundaries

Three planes, each with a distinct database posture:

- **Auth kernel** — inside `apps/web`, on the admin connection: identity,
  sessions, memberships. Identity tables are _revoked_ from the application
  role ([ADR-0004](https://github.com/achref-soua/helio/blob/main/docs/adr/0004-better-auth-kernel.md)),
  so a compromised tenant path can never read credentials, SSO secrets, or
  SCIM tokens.
- **Domain plane** — the `helio_app` role, with row-level security _forced_:
  everything tenant-owned. `forTenant()` is the only path to it
  ([ADR-0003](https://github.com/achref-soua/helio/blob/main/docs/adr/0003-postgres-rls-tenancy.md)).
- **Public edge** — `apps/api`: bearer-authenticated, rate-limited, idempotent,
  problem+json
  ([ADR-0008](https://github.com/achref-soua/helio/blob/main/docs/adr/0008-hono-openapi-gateway.md)).

This is why cross-tenant access is impossible at the database, not merely
filtered in application code. See [Tenancy](/docs/concepts/tenancy) for the
runtime mechanics.

## Durable execution

Every journey runs as one Temporal workflow per enrolled contact. Waits are
real durable timers, not rows polled by a cron — a worker can be killed with
`kill -9` mid-wait and, on restart, the journey resumes exactly where it left
off with no double-sends. Per-recipient send rows make every delivery
idempotent. The rationale is in
[ADR-0005](https://github.com/achref-soua/helio/blob/main/docs/adr/0005-temporal-orchestration.md)
and [ADR-0012](https://github.com/achref-soua/helio/blob/main/docs/adr/0012-journey-execution.md).
