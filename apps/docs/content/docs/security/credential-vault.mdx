---
title: Credential vault
description: How Helio stores provider credentials encrypted at rest, what the masks mean, and how key rotation works.
---

Helio sends through **your** providers — your SMTP relay or email API, your
Twilio account, your WhatsApp number, your AI key. Those credentials are
stored per organization in Postgres, sealed with AES-256-GCM under the
deployment's `HELIO_ENCRYPTION_KEY`. They are entered once in
**Settings → Provider credentials**, verified with a read-only connectivity
check, and never shown again: every read returns a mask built from the last
four characters captured at write time.

## What protects a stored secret

| Layer              | Guarantee                                                                                                                            |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------ |
| Encryption         | AES-256-GCM envelopes (`enc:v1:…`); the key never lives in the database.                                                             |
| Binding            | The cipher's AAD ties each value to its organization, row, and field — an envelope copied onto another row fails authentication.     |
| Row-level security | Credential rows are tenant-isolated like every domain table; the app role only resolves its own organization's rows.                 |
| Read paths         | API reads structurally exclude the sealed column; masks come from write-time metadata. Decryption happens server-side, at send time. |
| Audit              | Every create, update, verify, and delete is audit-logged with the kind and name only.                                                |

The same envelope also seals integration signing secrets (Shopify) and DKIM
private keys. Rows written before the vault existed remain readable and are
sealed in place by the rotation walk below.

## The deployment key

`HELIO_ENCRYPTION_KEY` is base64 of exactly 32 random bytes. The installer
generates it; manual deployments can mint one with:

```bash
openssl rand -base64 32
```

Keep it with your other deployment secrets — **a database backup without
this key cannot reveal stored credentials, but you also cannot recover them
without it.** Re-entering credentials is the recovery path if the key is
lost.

## Rotating the key

1. Generate a new key. Set `HELIO_ENCRYPTION_KEY` to it and
   `HELIO_ENCRYPTION_KEY_PREVIOUS` to the old key, then restart.
2. Run the re-encrypt walk (the `helio` CLI wraps this as `helio rotate-key`):

   ```bash
   DATABASE_ADMIN_URL=… HELIO_ENCRYPTION_KEY=… HELIO_ENCRYPTION_KEY_PREVIOUS=… \
     pnpm --filter @helio/db exec tsx scripts/rotate-encryption-key.ts
   ```

   The walk is idempotent and resumable: each sealed value is re-encrypted
   under the active key, and legacy plaintext secrets are sealed on the way.

3. Remove `HELIO_ENCRYPTION_KEY_PREVIOUS` and restart.

While both variables are set, every reader accepts either key (envelopes
carry a key fingerprint), so rotation needs no downtime.

## Cross-language guarantee

The envelope format is implemented twice — TypeScript (Web Crypto) and
Python (`cryptography`) for the intelligence service — and both replay the
same committed test vectors, so the byte layout cannot drift between the
two halves of the platform.
