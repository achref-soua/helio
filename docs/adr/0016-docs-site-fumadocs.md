# ADR-0016: Documentation site on Fumadocs

**Status:** accepted · 2026-06-09

## Context

The brief calls for a first-class documentation site (`apps/docs`) covering
concepts, self-hosting, configuration, every feature, the API and webhooks, the
SDKs, the MCP server, the migration guides, and an architecture deep-dive — and
it grades docs as heavily as code. The repository already README-documents the
product and keeps long-form docs (ADRs, architecture, SSO/SCIM, the REST guide)
as Markdown under `docs/`. We need a browsable, searchable site without
introducing a second framework or runtime to maintain.

Candidates: Docusaurus and VitePress (separate React/Vue toolchains), Nextra and
Fumadocs (both Next.js), and Astro Starlight.

## Decision

Build the docs site with **Fumadocs** (`fumadocs-ui` 16.9.3 + `fumadocs-core` +
`fumadocs-mdx` 15) on **Next.js 16 + React 19 + Tailwind v4** — the exact stack
the dashboard already runs. Content is authored as **MDX** under
`content/docs`, compiled by `fumadocs-mdx` into a generated `.source` (git-ignored,
regenerated by a `postinstall`), and served by an App-Router catch-all with a
static, build-time search index.

Fumadocs wins because it reuses our toolchain wholesale: the same pnpm workspace,
the shared `@helio/config` ESLint/TypeScript presets, Prettier, and the Turbo
`build`/`lint`/`typecheck` tasks. Its peer set (`next@16.x`, `react@^19.2`)
matches our pins exactly, so there is no version rabbit hole. Pages use explicit
route-param types rather than Next's generated `PageProps` globals, so the app
type-checks cold without a prior build (the local merge gate doesn't build).

Architecture diagrams stay as Mermaid in the repository Markdown (GitHub renders
them); the docs site links to them rather than bundling a client-side Mermaid
renderer. That keeps the docs build lean — a deliberate choice given the
project's memory-constrained build host — without sacrificing the written
deep-dive.

## Consequences

Documentation now has a real home that lints, type-checks, and builds in the
same pipeline as the rest of the monorepo, and a writer adds a page by dropping
an MDX file. The trade-off is one more Next.js app to build; it is excluded from
the `verify` gate's heavy steps (no test/build there) and only built on demand,
so it doesn't slow day-to-day iteration. Deployment and containerization of the
site, plus live-rendered diagrams, are intentionally deferred to the v1 platform
milestone alongside the public demo.
